http

HTTP 메세지에 모든 것을 전송

  • HTML, TEXT
  • IMAGE, 음성, 영상 파일 …
  • JSON, XML (API)
  • 거의 모든 형태의 데이터 전송 가능
  • 서버간에 데이터를 주고 받을 때도 대부분 HTTP 사용

server의 response는 status code와 함께 Response

수십개 존재하지만 그 중 중요한것.

기반 프로토콜

  • TCP : HTTP/1.1, HTTP/2

  • UDP : HTTP/3 (QUIC라는 계층 위에서 동작한다.)

  • 현재 HTTP/1.1 주로 사용 ⇒ 헤더에 많은 메타 데이터 들이 들어가 크기가 크다.

    • HTTP/2, HTTP/3 은 성능 개선의 차원으로 이해
    • HTTP/2 SPDY 프로토콜에서 파생된 HTTP/1.1보다 지연 시간을 줄이고, 응답 시간을 더 빠르게 하고, 멀티플렉싱, 헤더 압축, 서버 푸시, 요청의 우선순위 처리를 지원하는 프로토콜.
    • HTTP/2는 클라이언트가 요청 없이 서버가 바로 리소스를 푸시할 수 있다.

특징

… 단순하다!

1. Client-Server 구조

  • Request, Response 구조
  • client는 server에 요청을 보내고, 응답을 대기
  • server가 요청에 대한 결과를 만들어서 응답
  • server에는 데이터와 비즈니스로직에 집중, client는 UX, UI에 집중 → 서로 독립적으로 진화 가능하다.

2. 비연결성(Connectionless)

  • client가 request를 보내고, server가 response를 하면 그 연결을 끊어버리는 특징. 즉 연결을 유지하지 않는다.

  • HTTP 요청에 대한 응답은 일반적으로 초 단위 이하의 빠른속도이다.

  • 1시간 동안 수천명이 서비스를 사용해도 실제 서버에서 동시에 처리하는 요청은 수십개 이하로 매우 작다. (웹 브라우저에서 연속해서 검색 버튼을 누르지는 않는다.)

  • 서버 자원을 효율적으로 사용가능하다.

  • 한계

  • TCP/IP 연결을 새로 맺어야 함. → 3 way handshake 시간 추가

    • 웹 브라우저로 사이트를 요청하면, HTML 뿐만 아니라 static file등 수많은 자원이 다운로드
    • 현재는 HTTP 지속연결(Persistent Connections)로 문제 해결
    • HTTP/2, HTTP/3 에서 더 많은 최적화

3. 무상태 프로토콜(Stateless)

  • 통신이 끝나면 상태를 유지하지 않는 속성, 연결을 끊는 순간 상태정보는 유지하지 하지 않는 특성.
  • 최대한 stateless하게 server를 설계하는 것이 좋다.
  • Server가 Client의 상태를 보존하지 않는다.
  • stateful 할 경우, 서버에 문제가 생겼을 때, 처음 부터 요청을 다시 보내야 하는 문제가 생긴다. → 중간에 다른 서버로 바뀌면 안된다.
  • stateless의 경우 서버 한 곳에 문제가 생기면, 중계서버에 의해 다른 서버로 연결 가능하다.

⇒ scale out (수평 확장)에 유리하다. (무한한 증설 가능)

  • 모든 것을 무상태로 설계할수 있는 경우도 있고, 없는 경우도 있다.

예) 로그인

→ 상태 유지는 최소한만 사용한다.(만약 쿠키와 세션이 없다면, 사용자는 페이지 이동을 할 때 마다 로그인을 해야할 수도 있다.

… 서버 개발자들이 어려워하는 업무

  • 정말 같은 시간에 딱 맞추어 들어오는 대용량 트래픽
  • Message를 이용하여 통신

HTTP message의 구조

start-line 시작라인

header (헤더)

empty line (CRLF)

message body

Request

예시)

GET /search?q=hello&hl=ko HTTP/1.1

HOST: www.google.com

Response

예시)

HTTP/1.1 200 OK

Content-Type : text/html

…

구조별 특징

1. start-line

  • request-line 과 status-line이 들어감.
  • request-line = method SP(공백) request-target SP HTTP-version CRLF(엔터)
    • HTTP method (GET)
    • 요청 대상 (/search?q=hello&hl=ko)
    • HTTP Version (HTTP/1.1)
  • status-line = HTTP-version SP status-code SP reason-phrase CRLF
    • HTTP Version (HTTP/1.1)
    • status-code (200)
    • reason-phrase(OK) : 사람이 이해할 수 있는 짧은 상태 코드 설명 글

2. Header

  • header-field = field-name “:” OWS field-value OWS (OWS: 띄어쓰기 허용)

  • field-name은 대소문자 구분 없음

  • 용도

    • HTTP 전송에 필요한 모든 부가정보
    • 표준 헤더가 너무 많음
    • 필요시 임의의 헤더 추가 가능

3. Message Body

  • 실제 전송할 데이터
  • HTML 문서, 이미지, 영상, JSON 등등 byte로 표현할 수 있는 모든 데이터 전송 가능

HTTP Method

특징

1. safe

  • 호출해도 리소스를 변경하지 않는다. → GET

2. 멱등(idempotent)

서버에게 동일한 요청을 여러 번 전송하더라도 동일한 응답이 돌아와야 한다는 성질

  • 몇번을 호출하든 항상 같은 값을 반환해야 한다.

  • 멱등 method

    • GET: 몇 번을 조회하든, 같은 결과가 조회된다.
    • PUT: 결과를 대체한다. 따라서 같은 요청을 여러번 해도 최종 결과는 똑같다.
    • DELETE: 결과를 삭제한다. 같은 요청을 여러번 해도 삭제된 결과는 똑같다.

    POST는 멱등성이 없다.(Non-idempotent)

  • 활용

    • 자동 복구 메커니즘
    • 서버가 timeout 등으로 정상 응답을 못 주었을 때, 클라이언트가 같은 요청을 해도 되는지에 대한 판단 근거

3. cacheability → 응답 결과 resource를 캐시해서 사용해도 되는가?

  • 응답 결과 리소스를 캐시해서 사용해도 되는가?

  • GET, HEAD, POST, PATCH 캐시 가능

  • 실제로는 GET, HEAD 정도만 사용한다.

    → POST, PATCH는 본문 내용까지 캐시 키로 고려해야하는데, 구현이 쉽지 않음.

4. 안전

  • 호출해도 리소스를 변경하지 않는 method를 안전(Safe)하다고 하다.

종류

GET

  • Get은 서버로부터 정보를 조회하기 위한 메소드
  • 서버에 전달하고 싶은 데이터는 query(query parameter, query string)을 통해서 전달
  • Idempotent

POST

  • Message-body에 데이터를 담아 서버에 Data를 처리하도록 요청.
  • 리소스를 생성/변경하기 위한 메소드. 리소스를 처리하는 모든 요청은 POST 요청으로 보낼 수 있다.
  • Non-idempotent

사용유형

  1. 주로 전달된 데이터로 신규 리소스 등록, 프로세스 처리에 사용한다.
    • 서버가 식별하지 않은 새로운 리소스
  2. 요청 데이터 처리
    • 데이터 생성, 변경 뿐만 아니라 프로세스를 처리 해야 하는 경우 ex) 주문에서 결제 완료 → 배달 시작 → 배달 완료와 같이 프로세스의 상태를 변경하는 경우.
    • POST의 결과로 새로운 resource가 생성되지 않을 수도 있다.
  3. 다른 메서드로 처리하기 애매한 경우, POST를 쓸 수 있다.

PUT

  • 리소스를 대체
    • 리소스가 있으면 완전히 대체 → 수정 보다는 갈아치운다는 느낌
    • 리소스가 없으면 생성
    • 덮어쓰기 라고 생각
  • (중요!) 클라이언트가 리소스를 식별
    • 클라이언트가 리소스 위치를 알고 URI 지정 → POST와의 차이. POST는 서버에서 리소스 위치 알고, 요청에 대한 로직을 진행 후에, 위치를 반환해줌.

PATCH

  • 리소스 부분 변경

DELETE

  • 리소스 제거

  • 요청에 message-body가 없음.

  • HEAD : GET과 동일하지만 메세지 부분을 제외하고, 상태 줄과 헤더만 반환. 따라서 HTTP method 응답 중에 body가 없는 요청.

  • OPTIONS : 대상 리소스에 대한 통신 가능 옵션(메소드)을 설명 (주로 CORS에서 사용)

  • CONNECT : 대상 자원으로 식별되는 서버에 대한 터널을 설정

  • TRACE : 대상 리소스에 대한 경로를 따라 메시지 루프백 테스트를 수행

상태코드(Status Code)

클라이언트가 보낸 요청의 처리 상태를 응답에서 알려주는 기능

  • 1xx(informational) : 요청이 수신되어 처리중

2xx(Successful) : 요청 정상 처리

  • 200 : Success
  • 201 : Create Success
  • 202 : Accepted : 요청이 접수되었으나 처리가 완료되지 않았음. ex) 배치 처리
  • 204 : No Content : 서버가 요청을 성공적으로 수행했지만, 응답 페이로드 본문에 보낼 데이터가 없음.
    • save 버튼의 결과로 아무 내용이 없어도 된다.

3xx(Redirection) : 요청을 완료하려면 추가 행동이 필요

Redirection

  • 영구 리다이렉션 - 특정 리소스의 URI가 영구적으로 이동
    • ex) /members → users
    • ex) /event → /new-event
  • 일시 리다이렉션 - 일시적인 변경
    • 주문 완료 후 주문 내역 화면으로 이동
    • PRG: Post/Redirect/Get
  • 특수 리다이렉션 - 결과 대신 캐시를 사용

영구적 리다이렉션

  • 301 : Redirect (Moved Permanently)
    • 리다이렉트 시 요청 메소드가 GET으로 변하고, 본문이 제거될 수 있다.(MAY)
  • 308 : Permanent Redirect
    • 301과 기능은 같음
    • 리다이렉트시 요청 메서드와 본문 유지(처음 POST를 보내면 리다이렉트도 유지)
    • 실무에서는 301로 처리하는 경우가 많다. → URI가 바뀌면서, 제공해야 하는 data format이 바뀌기 때문이다.

일시적 리다이렉션

  • 302 : Found
    • 리다이렉트시 요청 메소드가 GET으로 변하고, 본문이 제거될 수 있음(MAY)
  • 307 : Temporary Redirect
    • 302와 기능은 같음.
    • Redirect시 요청 메소드와 본문 유지(요청 메소드를 변경하면 안된다. MUST NOT)
  • 303 : See Other
    • 302와 기능은 같음.
    • Redirect시 요청 메소드가 GET으로 변경되야 함.

(역사) 처음 302 스펙의 의도는 HTTP 메소드를 유지하는 것.

그런데 웹 브라우저들이 대부분 GET으로 바꾸어 버림.

그래서 모호한 302를 대신하는 명확한 307, 303이 등장함.

현실

  • 307, 303을 권장하지만, 현실적으로 이미 많은 애플리케이션 라이브러리들이 302를 기본값으로 사용한다.
  • 자동 리다이렉션 시에 GET으로 변해도 되면, 그냥 302를 사용해도 크게 문제 없음.

PRG: POST/Redirect/GET

  • POST로 상품 주문후에 새로고침을 했는데, 만약 POST 요청이 들어가면 중복 주문이 될 수 있다.

→ POST로 주문후에 주문 결과 화면을 GET 메서드로 Redirect

→ 새로고침 해도 결과 화면을 GET으로 조회

→ 중복 주문 대신, 결과 화면만 중복해서 GET 요청

기타 리다이렉션

  • 300 : Multiple Choices: 안 쓴다.
  • 304 : Not Modified
    • 캐시를 목적으로 사용
    • 클라이언트에게 리소스가 수정되지 않았음을 알려준다. 따라서 클라이언트는 로컬 PC에 저장된 캐시를 재사용한다.(캐시로 리다이렉트 한다.)
    • 304 응답은 응답에 메세지 바디를 포함하면 안된다. (로컬 캐시를 사용해야 하므로)
    • 조건부 GET, HEAD 요청시 사용

4xx(Client Error) : 클라이언트 오류, 잘못된 문법등으로 서버가 요청을 수행할 수 없음.

  • 오류의 원인이 클라이언트에 있음.

  • 재시도를 해도 실패한다. (5xx 은 서버의 문제이므로, 요청을 지속하면 성공할 수 있다.)

  • 400 : Bad Request

    • 요청 구문, 메세지 등등 오류
    • 클라이언트는 요청 내용을 다시 검토하고, 보내야함.
    • ex) 요청 파라미터, API 스펙이 안 맞을 때.

  • 401 : Unauthorized

    • 인증되지 않음.

    • 401 오류 발생시 응답에 WWW-Authenticate 헤더와 함께 인증방법을 설명

    • (참고)

      • 인증 : 본인이 누구인지 확인(로그인)
      • 인가 : 권한부여
      • 오류 메시지가 Unauthorized 이지만 인증되지 않음.

  • 403 : Forbbiden Error

    • 서버가 요청을 이해했지만, 승인을 거부함.
    • 주로 인증 자격 증명은 있지만, 접근 권한이 불충분한 경우.

  • 404 : Not found error (요청을 보낸 곳의 resource가 존재하지 않는다.)

    • 요청 리소스가 서버에 없음.
    • 또는 클라이언트가 권한이 부족한 리소스에 접근할 때 해당 리소스를 숨기고 싶을 때

  • 405

5xx(Server error) : 서버 오류, 서버가 정상 요청을 처리하지 못함.

  • 서버 내부 문제로 오류 발생

  • 애매하면 500 error

  • 500 : Internal Server Error

  • 503 : Service Unavailable

    • 서버가 일시적인 과부하 또는 예정된 작업으로 잠시 요청을 처리할 수 없음.
      • Retry-After 헤더 필드로 얼마뒤에 복구되는지도 보낼수 있음.

  • 모르는 상태코드가 나온다고 하면?

  • client는 상위 상태코드로 해석해서 처리

  • 미래에 새로운 상태 코드가 추가되어도 클라이언트를 변경하지 않아도 됨.

Client에서 Server로 Data 전송

데이터 전달 방식

1. 쿼리 파라미터를 통한 데이터 전송

  • GET
  • 주로 정렬 필터(검색어)

2. 메시지 바디를 통한 데이터 전송

  • POST, PUT, PATCH
  • 회원 가입, 상품 주문, 리소스 등록, 리소스 변경

4가지 유형

1. 정적 데이터 조회

  • GET
  • 이미지 정적 텍스트
  • 쿼리 파라미터 없이 리소스 경로로 단순하게 조회 가능

2. 동적 데이터 조회 with query parameter

  • GET
  • query parameter를 사용하여 데이터 전달
  • 게시판 목록에서 정렬 필터(검색어)
  • 조회 조건을 줄여주는 필터, 조회 결과를 정렬하는 정렬 조건에 주로 사용

3. HTML Form 데이터 전송

  • HTML의 form 태그를 이용

  • GET, POST 2가지 요청

  • 회원 가입, 상품주문, 데이터 변

  • 쿼리 파라미터와 동일한 방식으로 데이터 전송

  • content-type : application/x-www-form-urlencoded 사용

    • form의 내용을 메시지 바디를 통해서 전송(key=value, 쿼리 파라미터 형식)
    • 전송 데이터를 url encoding 처리
    • ex) abc 김 → abc%EA%B9%80

  • GET 전송도 가능

  • 만약 파일과 같은 binary 데이터를 전송하고 싶을 때는, content-type : multipart/form-data

  • 다른 종류의 여러 파일과 폼의 내용 함께 전송 가능

  • HTML Form 전송은 GET, POST만 지원

4. HTTP API를 통한 데이터 전송

  • 회원가입, 상품 주문, 데이터 변경
  • 서버 to 서버, 앱 클라이언트, 웹 클라이언트
  • server to server
  • 앱 클라이언트
  • 웹 클라이언트. JS를 통한 통신(AJAX)
  • POST, PUT, PATCH : 메세지 바디를 통해 데이터 전송
  • GET : 조회, 쿼리파라미터로 데이터 전달
  • Content-Type: application/json을 주로 사용 (사실상 표준)
    • TEXT, XML, JSON

API URI 설계

가장 중요한 것은 Resource 식별

아래 예시에서, 행위를 제외한 회원 만이 Resource!

  • 회원 등록 /members/{id}
  • 회원 목록 조회 /members
  • 회원 수정 /members/{id}
  • 회원 삭제 /members/{id}

POST - 신규 자원 등록 특징

  • 클라이언트는 등록될 리소스의 URI를 모른다.

  • 콜렉션(collection)

    • 서버가 관리하는 리소스 디렉토리
    • 서버가 리소스의 URI를 생성하고 관리
    • 여기서 컬렉션은 /members

  • 서버가 새로 등록된 리소스 URI를 생성해준다.

    • HTTP/1.1 201 Created
    • Location: /members/100

  • 클라이언트는 등록될 리소스의 URI를 모른다.

    • 회원 등록/members → POST
    • POST /members

PUT으로 신규자원 등록 할 때 특징

  • 클라이언트가 리소스 URI를 알고 있어야 한다.

  • 파일 등록 /files/{filename} → PUT

  • PUT /files/star.jpg

  • 클라이언트가 직접 리소스의 URI를 지정한다.

  • 스토어(Store)

    • 클라이언트가 관리하는 리소스 저장소
    • 클라이언트가 리소스의 URI를 알고 관리
    • 여기서 스토어는 /files

HTML FORM 사용

  • HTML FORM은 GET, POST만 지원
  • GET, POST만 지원하므로 제약이 있음.
  • 이런 제약을 해결하기 위해 동사로 된 리소스 경로 사용. 이를 Control URI 라고 함.
  • POST의 /new, /edit, /delete가 컨트롤 URI
  • HTTP method로 해결하기 애매한 경우 사용 (HTTP API 포함) → 실무에서 HTTP API로 해결할 수 없는 경우에 사용

일반적인 HTTP 헤더

표현(Representation)

Content-Type: 표현 데이터의 형식

  • 미디어 타입, 문자 인코딩
    • text/html; charset=utf-8
    • application/json
    • image/png

Content-Encoding: 표현 데이터의 압축 방식

  • 표현 데이터를 압축하기 위해 사용
  • 데이터를 전달하는 곳에서 압축 후 인코딩 헤더 추가
  • 데이터를 읽는 쪽에서 인코딩 헤더의 정보로 압축 해제
    • gzip
    • deflate
    • identity

Content-Language : 표현 데이터의 자연 연어

  • 표현 데이터의 자연언어를 표현
    • ko
    • en
    • en-US

Content-Length : 표현 데이터의 길이

  • byte 단위
  • Transfer-Encoding(전송 코딩)을 사용하면 Content-Length를 사용하면 안됨.
  • 표현 헤더는 전송, 응답 둘다 사용

협상 (Contents Negotiation)

클라이언트가 선호하는 표현 요청

  • Accept: 클라이언트가 선호하는 미디어 타입 전달

  • Accept-Charset : Client가 선호하는 문자 인코딩

  • Accept-Encoding: 클라이언트가 선호하는 압축 인코딩

  • Accept-Language: 클라이언트가 선호하는 자연 언어

  • 협상 헤더는 요청시에만 사용

협상과 우선순위

  • Quality Values(q)

  • 0 ~ 1, 클수록 우선 순위

  • 생략하면 1

  • Accept-Language: ko_KR, ko;q=0.9,en-US;q=0.8,en:q=0.7

  • 우선 순위 (구체적인 것이 우선한다.)

  1. text/plain;format=flowed
  2. text/plain
  3. text/*
  • 구체적인 것을 기준으로 미디어 타입을 맞춘다.
    Accept: text/*;q=0.3, text/html;q=0.7, text/html;level=1,
  • text/html;level=2;q=0.4, /*;q=0.5

전송 방식

1. 단순 전송

Content-Length 지정

2. 압축 전속

Content-Encoding:gzip

Content-length: 521

3. 분할 전송

Transfer-Encoding: chunked

Content-Length는 예상할 수 없으므로 넣지 않는다.

4. 범위 전송

Range, Content-Range

  • Length의 범위를 지정하여 전송
  • Content-Range: bytes 1001-2000/2000

쿠키 관련

why cookie?

  • HTTP는 무상태 프로토콜이니까 …

  • 쿠키 미사용 하면, 모든 요청에 사용자 정보 포함? → 보안 문제

  • 로그인을 성공하면,

Set-Cookie: user=홍길동

서버는 다음과 같이 쿠키를 Header에 같이 넣어서 보내준다.

그러면 웹 브라우저는 쿠키 저장소에, 쿠키를 저장 해준다.

웹 브라우저는 다음 부터 요청을 보낼 때,

GET /welcome HTTP/1.1

Cookie: user=홍길동

다음과 같이 cookie를 같이 보낸다.

쿠키

  • 사용처

    • 사용자 로그인 세션 관리
    • 광고 정보 트래킹

  • 쿠키 정보는 항상 서버에 전송됨

    • 네트워크 트래픽 추가 유발
    • 최소한의 정보만 사용(세션 id, 인증 토큰)
    • 서버에 전송하지 않고, 웹 브라우저 내부에 데이털흘 저장하고 싶으면 웹 스토리지 참고

  • 주의!

    • 보안에 민감한 데이터는 저장하면 안된다.

  • 생명주기

    • Expires, max-age
    • Set-Cookie: expires=Sat, 26-Dec-2020 04:39:21 GMT
      • 만료일이 되면 쿠키 삭제
    • Set-Cookie: max-age=3600 (3600초)
      • 0이나 음수를 지정하면 쿠키 삭제
    • 세션 쿠키: 만료 날짜를 생략하면 브라우저 종료시 까지만 유지
    • 영속 쿠키 : 만료 날짜를 입력하면 해당 날짜까지 유

  • Domain

  • domain=example.org

  • 명시: 명시한 문서 기준 도메인 + 서브 도메인 포함

    • domain=example.org를 지정해서 쿠키 생성
    • example.org + dev.example.org도 쿠키 접근

  • 생략 : 현재 문서 기준 도메인만 적용

    • example.org에서 쿠키를 생성하고 domain지정을 생략

  • 경로 Path

    • 이 경로를 포함한 하위 경로 페이지만 쿠키 접근
    • 일반적으로 path=/ 루트로 지정

    ex)

    • path=/home

    → /home , /home/level1 가능

    → /home/level1/level2 불가능

    • Secure, HttpOnly, SameSite

  • Secure

    • 쿠키는 http, https를 구분하지 않고 전송
    • Secure를 적용하면 https인 경우에만 전송

  • HttpOnly

    • Xss 공격방지
    • JS에서 접근 불가
    • HTTP 전송에만 사용

  • SameSite

    • XSRF 공격방지
    • 요청 도메인과 쿠키에 설정된 도메인이 같은 경우만 쿠키 전송

캐시

  • 캐시 덕분에 캐시 가능 시간동안 네트워크를 사용하지 않아도 된다.

  • 비싼 네트워크 사용량을 줄일 수 있다.

  • 브라우저 로딩 속도가 매우 빠르다.

  • 빠른 사용자 경험

  • cache-control: max-age=60

캐시 시간 초과

  • 캐시 유효시간이 초과하면, 서버를 통해 데이터를 다시 조회하고, 캐시를 갱신한다.

  • 이때 네트워크 다운로드가 발생한다.

  • 캐시 만료후에도 서버에서 데이터를 변경하지 않음.

  • 생각 해보면 데이터를 전송하는 대신에 저장해두었던 캐시를 재사용 할 수 있다.

  • 클라이언트의 데이터와 서버의 데이터가 같다는 사실을 확인할 수 있는 방법이 필요하다..

검증 헤더와 조건부 요청

  • 검증헤더
    • 캐시 데이터와 서버 데이터가 같은지 검증하는 데이터
    • Last-Modified, ETag

검증 헤더 추가

  • Last-Modified를 헤더에 추가
  • if-modified-since(조건부 요청) : 이후에 데이터가 수정되었으면?
    • 서버의 데이터가 미변경(갱신된게 없음)
      • 304 Not Modified 응답
      • HTTP Body가 없이 Header meta정보만 응답한다.
    • 데이터가 변경
      • 200 OK, 모든 데이터 전송(Body 포함)

결과적으로 네트워크 다운로드가 발생하지만 용량이 적은 헤더 정보만 다운로드 한다.

단점

  • 1초 미만 단위로 캐시 조정이 불가능
  • 날짜 기반의 로직 사용
  • 데이터를 수정해서 날짜가 다르지만, 같은 데이터를 수정해서 데이터 결과가 똑같은 경우
  • 서버에서 별도의 캐시 로직을 관리하고 싶은 경우(ex. 스페이스나 주석처럼 크게 영향이 없는 변경에서 캐시를 유지하고 싶은 경우)

⇒ Etag 사용

Etag, if-None-Match

  • Etag(Entity Tag)
  • 캐시용 데이터에 임의의 고유한 버전 이름을 달아둠.
  • 데이터가 변경되면 이 이름을 바꾸어서 변경함(Hash를 다시 생성)
  • 진짜 단순하게 ETag만 보내서 같으면 유지, 다르면 다시 받기
  • 캐시 제어 로직을 서버에서 완전히 관리
  • 클라이언트는 단순히 이 값을 서버에 제공(클라이언트는 캐시 메커니즘을 모른다.)

헤더 정리

  • Cache-Control: 캐시 지시어 → 캐시 제어

    • Cache-Control:max-age
      • 캐시 유효 시간, 초 단위
    • Cache-Control:no-cache
      • 데이터는 캐시해도 되지만, 항 상 origin 서버에 검증하고 사용
    • Cache-Control:no-store
      • 데이터에 민감한 정보가 있으므로 저장하면 안된다. (메모리에서 사용하고 최대한 빨리 삭제)

  • Pragma: 캐시 제어(하위 호환)

    • HTTP 1.0 하위 호환

  • Expires: 캐시 만료일 지정(하위 호환)

    • 캐시 만료일을 정확한 날짜로 지정
      • HTTP 1.0부터 사용
      • 현재는 Cache-Control:max-age 권장 → 함께 사용되면, Expires 무시

  • 검증헤더 (Validator)

    • ETag
    • Last-Modified

  • 조건부 요청 헤더

    • if-Match, if-None-Match: ETag 값 사용
    • If-Modified-Since, If-Unmodified-Since: Last-Modified 값 사용

프록시 캐시 도입

  • Origin Server 직접 접근 ⇒ 생각보다 느리다. ⇒ 프록시 캐시 서버 도입

Cache-Control - 기타

  • Cache-Control:public
    • 응답이 pubilc 캐시에 저장 되어도 됨.
  • Cache-Control:private
    • 응답이 해당 사용자만을 위한 것임, private 캐시에 저장해야 함.
  • Cache-Control:s-maxage
    • 프록시 캐시에만 정용되는 max-age
  • Age: 60 (HTTP 헤더)
    • origin server에서 응답 후 프록시 캐시 내에 머문 시간

캐시 무효화

  • 확실한 캐시 무효화 응답을 하고 싶을 때는

  • Cache-Control: no-cache, no-store, must-revalidate

  • Cache-Control: no-cache

    • 캐시 가능, 항상 오리진 서버에 검증
    • 만약 오리진 서버와의 네트워크가 순간 단절된다면, 옛날 데이터를 보여주면서, 200 OK를 보내는 겨경우가 있다.

  • Cache-Control: no-store

    • 데이터에 민감한 정보가 있으므로 저장하면 안됨.(메모리에서 사용하고 최대한 빠르게 삭제)

  • Cache-Control: must-revalidate

    • 캐시 만료 후 최초 조회시 원 서버에 검증
    • 원 서버 접근 실패의 경우 항상 오류가 발생한다.
    • 504 Gateway Timeout
    • 매우 중요한 정보, 돈과 관련된 경우

  • Pragma: no-cache

    • HTTP 1.0 하위 호환(하위 버전까지 생각)

HTTP BODY(메세지 본문)

  • 메시지 본문(message body)을 통해 표현 데이터 전달
  • 메시지 본문 = 페이로드(payload)
  • 표현은 요청이나 응답에서 전달할 실제 데이터
