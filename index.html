git

git config

git config --list : 전체 config list 확인

git config --global --list : global config list 확인

git config --global user.name <github username> : username 등록

git config --global user.email <github email> : email 등록

git config --unset [user.name](http://user.name) : 설정 삭제

git config --unset [user.email](http://user.email) : 설정 삭제

git의 3가지 Area

  • Working directory : 내가 작업하고 있는 프로젝트의 디렉토리
  • Staging Area : 커밋을 하기 위해 추가한 파일들이 모여있는 공간 (git add 명령어를 했을 때, 파일들이 저장되는 공간)

**Staging Area가 필요한 이유?**

  1. 일부분만 commit을 하고 싶을 때
  2. commit 취소를 할 때 용이
  3. 충돌을 해결할 때. ⇒ 작은 규모에서 충돌을 해결할 수 있다
  • Repository : 커밋들이 모여있는 저장소 (Local , Remote)

git의 4가지 File Status

  • Untracked : Working Directory에서 작업하는 파일이지만, Git으로 버전관리를 하지 않은 상태
  • Unmodified : git add 로 파일이 추가된 상태
  • Modified : 파일이 추가된 이후 파일이 수정되었을 때의 상태
  • Staged : Staging Area에 반영된 상태

기본적인 git 명령어

git init

  • git init : create an empty git repository, reinitialize an existing one

git clone

  • git clone <repoitory> : repository 복사
  • git clone <repository> -b : 특정 branch만 clone 받고 싶을 때

git add (Working directory 에서 staging area로 보내는 명령어)

  • git add <file name> : 파일 추가
  • git add . : 파일 전부 추가

git commit (Staging Area에서 Local Repository로 보내는 명령어)

  • git commit -m "message"
  • git commit --amend : 가장 최근 commit의 내용을 수정할 수 있다.

git remote

  • git remote : 원격서버에 연결 되었는지 확인.
  • git remote rm origin : 원격서버 삭제
  • git remote -v : 원격저장소 주소 확인
  • git remote add <remote name> <repository 주소> : 원격 저장소 연결 (remote name은 default가 origin>
  • git remote update : 모든 원격 브랜치를 업데이트 한다

git push & git pull

  • git pull origin (가지이름) : 로컬 저장소로 불러온다.

  • git push origin (가지이름) : 원격저장소로 보낸다.

  • git push -u origin main : 다음에 push 할 때, git push 만 입력해도 push가 된다.

  • git push origin --delete <branch name> : 로컬에서 원격 브랜치 제거

  • git push origin -u +<branch name> : 충돌 여부와 관계없이 강제로 push(사용 안하는게 좋음.)

git log

  • git log : git commit log를 확인할 수 있다.
  • git log --oneline : git log를 한 줄 단위로 확인할 수 있다.
  • git log --graph --decorate --oneline : git log를 graph 화 하여 보여준다.
  • git log --grep 검색어 : 특정 단어를 검색하며 보여주는 옵션

부가 명령어 (파일 변경 내용 확인, 파일 되돌리기, 임시저장 등등…)

git show

  • git show : 가장 최근 커밋 정보를 확인하는 명령어
  • git show <commit id> : 해당 commit id의 commit 정보를 확인

git diff - 내용 확인

  • git diff : commit된 파일상태와 현재 수정중인 상태 비교. add가 되었으면, 표시 안됨
  • git diff --staged : commit된 파일 상태와 add된 파일 상태 비교
  • git diff --cached : staged된 파일 상태에서, Staging Area(index) 영역과 Repository영역을 비교하며 변경사항을 표시한다.
  • git diff <commit id1> <commit id2> : commit 간의 파일 상태 비교
  • git diff HEAD HEAD^ : 가장 최근 커밋과 그 이전 커밋을 비교한다.
  • git diff <branch1> <branch2> : branch 간의 상태 비교.

git blame - 파일 변경 내용 확인

  • git blame <file name> : 해당 파일의 수정 이력을 볼 수 있다. (commit id, editor, 시간, commit message)

git stash : 아직 완료하지 않은 일을 stack에 저장

  • 기존에 하던 작업들을 Commit을 하지 않은 시점에서, 잠시 보관해두고 싶을 때 사용한다.

    1. Modified & Tracked
    2. Staged 된 파일(Staging Area에 있는 파일) : git add로 추가된 파일

  • git stash or git stash save : 작업하던 파일들을 stash로 보낸다.

  • git stash -m "" : stash 하면서 메세지를 저장할 수 있다.

  • git stash apply : stash 해놨던 작업을 다시 가져온다.

  • git stash list : stash 한 목록들을 확인할 수 있다.

  • git stash pop : stash에서 제거하면서, stash 한 내용 가져오기 (apply + drop 으로 이해)

  • git stash drop <stash id> : stash에서 제거, stash id 입력 안하는 경우, 가장 최근

  • git stash clear : stash 목록 전부 비우기

  • git stash branch <branch name> : stash에 있는 내용들을 포함하여 새롭게 branch 생성

git revert

  • git revert <commit id> : 돌아가고 싶은 commit id로 되돌아간다.

  • 돌아가고 싶은 commit 지점까지 순차적으로 revert 해준다.

  • git reset 과의 차이점?

    • git reset 의 경우 충돌의 가능성이 있다.
    • git revert 를 사용하면, 중간에 무슨문제가 있었는지, 기록을 할 수 있다.

git reset

  • git reset에는 3가지 option이 있다.

    1. git reset —soft <commit id> : commit 된 파일들을 staging area로 돌려 놓음.
    2. git reset —mixed <commit id> : commit된 파일들을 working directory로 돌려놓음. (add 하기 전 상태로)
    3. git reset —hard <commit id> : commit된 파일들 중 tracked 파일들을 working directory에서 삭제한다. (untracked 파일은 untracked 로 남는다.)

    HEAD~숫자 : 현재로부터 숫자 만큼의 커밋이 취소된다.

    HEAD^ : 가장 최근 커밋이 취소된다.

git restore

  • git restore <file name> : 특정 파일을 HEAD commit으로 복구
  • git restore --staged <file name> : 특정 파일을 unstaged 상태로 되돌린다.

git fetch

  • **git fetch**
    원격 저장소의 내용을 로컬 저장소로 받아오기는 합니다. 하지만 로컬 저장소는 내 컴퓨터에 저장이 되어 있을 뿐이지 우리가 우리가 실제로 작업하고 있는 Working Directory가 아닙니다. 즉, fetch 만으로는 직접 작업을 하거나 수정을 할 수가 없습니다.
  • git fetch -all : 모든 원격 브랜치를 업데이트 한다

git status

  • git status : 작업하고 있는 branch의 상태 변경을 확인.

Branch 관련

git branch

  • git branch : Local branch list를 보여준다.
  • git branch -a : Local, Remote branch list를 모두 보여준다.
  • git branch -r : Remote branch list를 보여준다.
  • git branch -v : branch 별로 가장 최근 update를 보여준다.
  • git branch -d <branch name> : branch 삭제

git switch

  • git switch <branch_name> : 작업하는 Branch 변경
  • git switch -C <branch_name> : 작업하는 branch 생성 후 변경

git checkout

  • git switch와 똑같지만, deprecated될 예정이므로, switch를 사용하자.
  • git checkout <branch name> : branch 이동
  • git checkout -b <branch name> : branch 생성 후 이동

Branch 합치기

git merge

  • 개별 브랜치에서의 작업을 마치고, 다른 브랜치로 합친다.
  • git merge develop

git rebase

  • merge와 결과는 같지만 commit history가 달라진다. rebase할 branch를 기준으로, 새롭게 project를 관리한다.
  • push 된 상황에서는 사용 안 하는게 좋다.

.gitignore

  • .gitignore 파일은 말 그대로 git 으로 하여금 특정 파일을 무시하여 commit에서 제외시키는 역할을 합니다. 보통 저장소의 최상위 디렉토리에 위치하고 있습니다.
  • .gitignore 에서 관리하는 파일은 보통 OS상에서 자동으로 생성하는 파일이나, 보안상 공개되어서는 안되는 파일, 공동 작업에 있어서 불필요한 것들로 이루어져 있습니다. ssh key, IAM pem 등은 절대 commit 되지 않도록 주의해야 합니다.
  • 통합 개발 환경에 따라 각 프레임워크에서 꼭 제외되어야 하는 파일들을 자동으로 설정해주기도 합니다. 만약 자동으로 생성해주지 않는다면 다음의 웹사이트에서 프레임워크 별로 제외해야되는 파일을 검색해서 사용하여도 무방합니다.
  • gitignore.io 에서 제외해야 되는 파일 예시를 쉽게 확인할 수 있다.

annotation 생성

  • @interface 를 이용하여, Annotation을 만들 수 있다.

@Target 을 이용하여, 어떤 형태로 Annotation을 사용할지 지정해 줄 수 있다.

  • ElementType 으로 어디 부분에서, Annotation으로 쓰일지 결정할 수 있다.

@Retention : Annotation의 lifecycle을 정하는 것.

  • RetentionPolicy
  1. RetentionPolicy.SOURCE : 소스 코드(.java)까지 남아 있는다. ⇒ ex) Lombok의 Getter, Setter 의 경우, 롬복이 바이트 코드를 생성 해서 넣어주므로, 바이트 코드에 어노테이션 정보가 필요 없다.
  2. RetentionPolicy.CLASS : 클래스 파일(.class)까지 남아 있는다. →CLASS가 필요한 이유는 Maven/Gradle로 받은 라이브러리와 같이 jar 파일에는 소스가 앚니라 class 파일만 포함되어 있기 때문이다.
  3. RetentionPolicy.RUNTIME : 런타임까지 남아있는다. ⇒ 런타임에 어노테이션 정보를 뽑아 쓸 수 있다. ex) @Controller, @Service, @Autowired

스프링이 올라오는 실행 중인 시점에 컴포넌트 스캔이 가능해야 하므로, RUNTIME 정책이 필요하다. (스프링도 내부적으로 Reflection 등을 활용하여 어노테이션이 사용된 요소들만 가져온다.)

REST = Architecture ( = Design Pattern)

  • 제약조건들의 집합
  • Client와 Server의 결함성을 줄이기 위한 가이드

REST

REpresentational

State

Transfer

1. Client Server Architecture

  • Client와 Server의 분리가 잘 되어 있어야한다.
  • 서로의 변화가 서로에게 영향을 주지 않는 형태가 되어야 한다.

2. Statelessness

  • 상태를 저장하지 않는다. → 원하는 기능을 위한 상태는 Client가 가지고 있어야 한다.

3. Cacheability

  • 캐쉬가 가능한지

4. Layered System

  • 서버는 계층구조를 갖고 있다.
  • 클라이언트는 서버에 도달하기까지의 과정을 알 필요 없다.

5. Uniformed Interface

  • 일관된 인터페이스
  • 자원을 나타내기 위한 인터페이스는 명확해야 한다.
  • HATEOAS(Hypermedia as the engine of application state)

6. Code on Demand(Optional

  • 일시적 기능의 확장
  • 사용가능한 코드를 응답으로 보내 사용자의 기능을 일시적으로 확장 시킬 수 있다.
