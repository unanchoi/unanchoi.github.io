REST = Architecture ( = Design Pattern)

  • 제약조건들의 집합
  • Client와 Server의 결함성을 줄이기 위한 가이드

REST

REpresentational

State

Transfer

1. Client Server Architecture

  • Client와 Server의 분리가 잘 되어 있어야한다.
  • 서로의 변화가 서로에게 영향을 주지 않는 형태가 되어야 한다.

2. Statelessness

  • 상태를 저장하지 않는다. → 원하는 기능을 위한 상태는 Client가 가지고 있어야 한다.

3. Cacheability

  • 캐쉬가 가능한지

4. Layered System

  • 서버는 계층구조를 갖고 있다.
  • 클라이언트는 서버에 도달하기까지의 과정을 알 필요 없다.

5. Uniformed Interface

  • 일관된 인터페이스
  • 자원을 나타내기 위한 인터페이스는 명확해야 한다.
  • HATEOAS(Hypermedia as the engine of application state)

6. Code on Demand(Optional

  • 일시적 기능의 확장
  • 사용가능한 코드를 응답으로 보내 사용자의 기능을 일시적으로 확장 시킬 수 있다.
